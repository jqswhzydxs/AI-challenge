# 接口测试参数清单

更新时间：2026-07-18

## 通用请求头

除登录接口外，建议都带上登录返回的 token：

```text
Authorization: Bearer 你的token
```

登录账号：

```json
{
  "username": "admin",
  "password": "123456"
}
```

## 3. 生产管理

### GET /api/production/orders

Query 参数分开填：

```text
pageNum=1
pageSize=10
date=
status=
```

也可以只填：

```text
pageNum=1
pageSize=10
```

### GET /api/production/schedules/{scheduleId}

不能直接发送 `/api/production/schedules/{scheduleId}`，必须替换成具体 ID：

```text
/api/production/schedules/4000000000000000101
```

### POST /api/production/schedule/generate

Body:

```json
{
  "scheduleDate": "2026-07-17",
  "planHorizon": 24,
  "planUnit": "hour",
  "dataGranularity": "1 hour",
  "objective": "MIN_EC",
  "constraints": {
    "elecCoefficient": 13.278524
  }
}
```

返回里的 `taskId` 可以继续查任务状态，`resultId` 可以查排产方案详情。

### POST /api/production/schedules/import-daily-plan

Body:

```json
{
  "timestamp": "2026-07-17 16:10:30",
  "plan_horizon": 24,
  "unit": "hour",
  "data_granularity": "1 minute",
  "elec_coefficient": 13.278524,
  "total_demand": 9.436921,
  "total_energy": 125.969628,
  "schedule": [
    {
      "hour": 0,
      "demand": 0.3791368692364509,
      "production": 0.15,
      "elec_forecast": 1.991779
    },
    {
      "hour": 1,
      "demand": 0.520875719376839,
      "production": 0.95,
      "elec_forecast": 12.614598
    }
  ]
}
```

注意：这个接口当前后端要求 `schedule` 必须 24 条。上面只是结构示例，真实测试建议直接复制算法组给的 `daily_plan_v3.2.json`。如果 JSON 里没有 `elec_forecast`，后端会用 `production * EC_optimized` 自动计算小时预测电耗。

## 4. 任务

### GET /api/tasks/{taskId}

不能直接发送 `/api/tasks/{taskId}`，必须替换成具体 ID：

```text
/api/tasks/4000000000000000001
```

如果刚调用了生成排产方案/生成能源方案/生成协同优化方案，也可以用返回的 `taskId`。

## 5. 能源管理

### GET /api/energy/realtime

Query 参数分开填：

```text
pageNum=1
pageSize=50
startTime=2026-07-17 00:00:00
endTime=2026-07-17 12:00:00
interval=15
```

`date/status/startDate/endDate` 可以留空。

### POST /api/energy/plan/generate

说明：算法组目前没有独立提供 `energy_plan.json`，这个接口当前用于联调演示，会根据请求生成一份后端派生/模拟能源运行方案。前端页面文案建议写“能源运行视图/能源运行方案展示”，不要写成“算法组独立输出能源方案文件”。

Body:

```json
{
  "planDate": "2026-07-17",
  "timeRange": "24h",
  "electricPriceMode": "PEAK_VALLEY",
  "objective": "MIN_ENERGY_COST",
  "constraints": {
    "steamDemand": 230,
    "maxBoilerLoad": 60
  }
}
```

返回的 `resultId` 可以用于查询能源方案详情。

### GET /api/energy/plans/{planId}

如果你刚调用过生成能源方案，用返回的 `resultId`。

示例路径参数：

```text
planId=后端返回的resultId
```

实际 URL：

```text
/api/energy/plans/后端返回的resultId
```

## 6. 实时调控

### POST /api/realtime-control/import

联调用：直接导入算法组给的 `realtime_control.json`。如果 JSON 里的 `timestamp` 只有 `HH:mm:ss`，必须在 Query 里带上控制日期。

URL:

```text
/api/realtime-control/import?controlDate=2026-07-17&sourceFileName=realtime_control.json
```

Body:

```json
{
  "timestamp": "01:00:00",
  "control": {
    "boiler_load": 30.01,
    "turbine_output": 10.178,
    "grid_purchase": 0,
    "power_factor_target": 0.95
  },
  "forecast": {
    "elec_next_5min": 4.182,
    "steam_next_5min": 0.505
  }
}
```

也支持数组或 `{ "data": [...] }` 这种汇总结构。重复导入同一天同一时间点时，后端会更新原记录，不会报唯一键冲突。

### GET /api/realtime-control/latest

无参数。

如果返回 404，说明 `mpc_realtime_control` 表里还没有数据，先调用上面的导入接口。

### GET /api/realtime-control/history

Query 参数分开填：

```text
pageNum=1
pageSize=10
startTime=2026-07-17 00:00:00
endTime=2026-07-18 00:00:00
```

## 7. 协同优化

### POST /api/optimize/joint/generate

说明：算法组目前没有独立提供 `joint_optimization.json`，这个接口当前用于联调演示，会基于已有排产方案和能源运行方案生成后端派生/模拟协同优化结果。前端页面文案建议写“协同优化评价/协同结果展示”，不要写成“算法组独立输出协同优化文件”。

需要已有排产方案 ID 和能源方案 ID。

Body:

```json
{
  "scheduleId": 4000000000000000101,
  "energyPlanId": 你生成能源方案后返回的resultId,
  "objective": "MIN_TOTAL_COST"
}
```

### GET /api/optimize/joint/{optimizeId}

用上一步返回的 `resultId`。

实际 URL：

```text
/api/optimize/joint/上一步返回的resultId
```
