# 生产-能源交互式优化平台前后端/算法数据接口约定（后端讨论稿）

> 适用项目：挑战杯”AI 赋能降本增效——创新型生产-能源交互式优化技术”  
> 适用对象：前端组、后端组、算法建模组  
> 版本：v0.6 讨论稿，2026-07-20 更新（算法组新版 daily_plan / realtime_control）  
> 维护责任：后端组维护接口结构、任务流程、落库口径；前端组确认页面字段；算法组确认输入输出字段、单位和计算口径。

---

## 1. 文档目标

本文档先把三组协作边界定清楚：

- 前端只调用后端 `/api/**` 接口。
- 后端统一负责参数校验、任务状态管理、算法调用、算法结果校验、数据库保存和统一响应。
- 算法组只面向后端提供稳定的 `/algorithm/**` 接口或固定 JSON 输出，不直接适配前端页面。

当前三方已确认：算法组现阶段只提供两类真实输出文件，分别是日级排产计划 `daily_plan` 和实时调控指令 `realtime_control`。能源运行方案、协同优化结果暂不作为算法组独立 JSON 文件交付，后端在联调阶段按“基于真实排产/实时调控数据的派生展示或模拟结果”提供接口，前端和 PPT 文案需避免写成算法组独立输出。

## 2. 总体调用流程

### 2.1 三方调用关系

```text
前端页面 -> 后端 /api/** -> 算法 /algorithm/**
前端页面 <- 后端 /api/** <- 算法 /algorithm/**
```

### 2.2 算法任务统一异步流程

排产、能源运行、协同优化等任务统一采用“创建任务 + 查询任务状态 + 查询结果详情”的流程。其中日级排产和实时调控是真实算法文件接入；能源运行方案和协同优化结果当前由后端基于已有排产、实时能源和实时调控数据派生计算，后续算法组扩展模型后可再切换为真实算法输出。

```text
1. 前端提交生成请求
2. 后端创建 taskId，状态置为 PENDING
3. 后端保存前端请求原始 JSON
4. 后端组装算法入参，调用算法服务
5. 后端保存算法请求和响应原始 JSON
6. 后端校验算法结果并落库
7. 前端轮询 GET /api/tasks/{taskId}
8. 任务 SUCCESS 后，前端根据 resultId 查询方案详情
```

后端暂定前端轮询间隔为 3-5 秒，具体由前端确认。

## 3. 通用规范

### 3.1 请求地址

| 调用方 | 被调用方 | 路径前缀 | 说明 |
|---|---|---|---|
| 前端 | 后端 | `/api/**` | 页面、登录、任务、报表、方案查询 |
| 后端 | 算法 | `/algorithm/**` | 排产、能源、协同优化、实时调控 |

### 3.2 认证方式

除登录接口外，前端请求后端接口需要携带 Token。

```http
Authorization: Bearer {token}
```

Token 失效时后端返回：

```json
{
  "code": 401,
  "message": "未登录或 Token 已失效",
  "data": null
}
```

### 3.3 统一响应格式

所有后端返回给前端的接口统一包一层 `code/message/data`。

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

失败响应：

```json
{
  "code": 602,
  "message": "算法返回结果校验失败",
  "data": null
}
```

### 3.4 通用状态码

| code | 含义 | 后端处理 |
|---:|---|---|
| 200 | 成功 | 正常返回数据 |
| 400 | 请求参数错误 | 参数缺失、格式错误、业务规则不满足 |
| 401 | 未登录或 Token 失效 | 前端跳转登录页 |
| 403 | 无权限 | 当前角色无权限访问 |
| 404 | 数据不存在 | 查询对象不存在或已删除 |
| 500 | 系统内部错误 | 后端未预期异常 |
| 600 | 算法服务调用失败 | 算法服务不可用、连接失败 |
| 601 | 算法任务超时 | 算法执行超过约定时间 |
| 602 | 算法返回结果校验失败 | 字段缺失、单位异常、时间序列不连续 |

### 3.5 分页规范

分页请求参数统一使用 query 参数。

```http
GET /api/production/orders?pageNum=1&pageSize=10
```

分页响应统一放在 `data` 内。

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "total": 100,
    "pageNum": 1,
    "pageSize": 10,
    "records": []
  }
}
```

### 3.6 时间、数值和单位

| 类型 | 约定 |
|---|---|
| 日期 | `yyyy-MM-dd`，如 `2026-07-10` |
| 日期时间 | `yyyy-MM-dd HH:mm:ss`，如 `2026-07-10 08:00:00` |
| 金额 | 默认单位 `元` |
| 比例 | 使用百分数数值，`96.5` 表示 `96.5%` |
| 小数 | 后端建议统一保留 2-4 位，前端展示可自行格式化 |

### 3.7.1 当前数据报告字段映射

当前 `数据报告.txt` 对应 GitHub `steel-industry-energy-dataset`，数据粒度为 15 分钟/点，总记录数 35040，缺失值数量 0，数据完整率 100%。

| 源字段 | 后端接口字段 | 说明 |
|---|---|---|
| `timestamp` | `timestamp` | 源数据为文本，后端入库时需要解析为 `yyyy-MM-dd HH:mm:ss` |
| `elec` | `electricityConsumption` | 15 分钟用电量，kWh |
| `steam` | `steamConsumption` | 蒸汽用量，单位需算法组确认 |
| `CO2_tCO2_` | `carbonEmissionTco2` | 碳排放，tCO2 |
| `Lagging_Current_Reactive_Power_kVarh` | `laggingReactivePowerKvarh` | 滞后无功电量 |
| `Leading_Current_Reactive_Power_kVarh` | `leadingReactivePowerKvarh` | 超前无功电量 |
| `Lagging_Current_Power_Factor` | `laggingPowerFactor` | 滞后功率因数 |
| `Leading_Current_Power_Factor` | `leadingPowerFactor` | 超前功率因数 |
| `NSM` | `nsm` | 日内秒数 |
| `week_status` | `weekStatus` | 工作日/周末 |
| `Day_of_week` | `dayOfWeek` | 星期 |
| `load_type` | `loadType` | 负荷类型 |

数据报告中时间范围显示为 `11-Jul-0018` 到 `12-Dec-2018 23:45:00`，起始年份和持续天数疑似解析异常。后端入库前需要统一清洗为合法日期时间，建议和算法组确认是否应理解为 `2018-01-01` 到 `2018-12-31` 或数据集原始日期范围。

### 3.7.2 日级排产模型输出字段

算法组已提供 `daily_plan.json`，用于未来 24 小时日级生产排产结果入库和前端展示。该模型由 132481 条真实历史数据回归拟合，电耗系数为 `14.00 kWh/吨`；算法组说明原始数据完成了 `15 分钟 -> 1 分钟` 粒度处理，但当前 JSON 的 `schedule` 数组按小时汇总，共 24 条记录。

| JSON 字段 | 后端接口字段 | 说明 |
|---|---|---|
| `timestamp` | `planStartTime` | 排产计划开始时间 |
| `plan_horizon` | `planHorizon` | 计划跨度，当前为 24 |
| `unit` | `planUnit` | 计划跨度单位，当前为 `hour` |
| `data_granularity` | `dataGranularity` | 模型底层数据粒度，当前为 `1 minute` |
| `EC_baseline` | `ecBaseline` | 优化前 EC 基准值，kWh/吨 |
| `EC_optimized` | `ecOptimized` | 优化后 EC 值，kWh/吨 |
| `EC_reduction` | `ecReduction` | EC 降低百分比 |
| `optimal_temperature` | `optimalTemperature` | 最优温度，℃ |
| `optimal_speed` | `optimalSpeed` | 最优速度 |
| `total_production` | `totalProduction` | 24 小时总排产量，吨 |
| `total_demand` | `totalDemand` | 24 小时总需求 |
| `total_energy` | `totalEnergy` | 24 小时预测总电耗，kWh |
| `schedule[].hour` | `hourIndex` | 第几个小时，0-23 |
| `schedule[].demand` | `demand` | 该小时需求 |
| `schedule[].production` | `production` | 该小时排产量，吨 |
| `schedule[].elec_forecast` | `elecForecast` | 新版 JSON 已移除该字段，后端按 `production × EC_optimized` 自动计算 |

后端入库时建议同时保存 `daily_plan.json` 原始报文，并拆分保存方案主表和小时明细表。

### 3.8 任务状态枚举

| 状态值 | 中文含义 | 使用场景 |
|---|---|---|
| `PENDING` | 待执行 | 后端已创建任务，尚未调用算法 |
| `RUNNING` | 执行中 | 算法正在计算或后端正在保存结果 |
| `SUCCESS` | 执行成功 | 算法完成且结果已保存 |
| `FAILED` | 执行失败 | 算法失败、超时或结果校验失败 |
| `CANCELED` | 已取消 | 用户取消任务 |
| `PUBLISHED` | 已发布 | 方案已发布给业务执行 |
| `EXECUTED` | 已执行 | 方案已完成执行 |

### 3.9 任务类型枚举

| taskType | 中文含义 |
|---|---|
| `PRODUCTION_SCHEDULE` | 生产排产任务 |
| `ENERGY_PLAN` | 能源运行优化任务 |
| `JOINT_OPTIMIZATION` | 生产-能源协同优化任务 |
| `REALTIME_CONTROL` | 实时调控任务 |
| `REALTIME_MPC` | MPC 实时调控任务 |

### 4.4 生成生产排产方案

```http

```

请求：

```json
{
  "scheduleDate": "2026-07-10",
  "planHorizon": 24,
  "planUnit": "hour",
  "dataGranularity": "1 minute",
  "objective": "MIN_COST",
```

响应：

```json
{
  "code": 200,
  "message": "排产任务已创建",
  "data": {
    "taskId": 5001,
    "taskType": "PRODUCTION_SCHEDULE",
    "status": "PENDING"
  }
```

后端说明：

- 后端创建任务后保存请求参数原始 JSON。

- `dataGranularity` 表示算法底层数据粒度；当前排产结果仍按小时汇总展示。

待算法组确认：

- `objective` 可选值是否只包含 `MIN_COST`，是否还需要 `MIN_TIME`、`MAX_OUTPUT`。
- 后续是否输出 1 分钟级明细。如果只输出小时汇总，后端按小时入库和展示。

### 4.6 实时能源数据

```http
GET /api/energy/realtime
```

请求参数：

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `startTime` | string | 是 | 开始时间 |
| `endTime` | string | 是 | 结束时间 |
| `interval` | integer | 否 | 返回粒度，单位分钟；当前数据集基础粒度为 15 |

响应：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "timeInterval": 15,
    "points": [
      {
        "timestamp": "2026-07-10 08:00:00",
        "electricityConsumption": 86.7,
        "steamConsumption": 3.8,
        "carbonEmissionTco2": 0.12,
        "laggingReactivePowerKvarh": 12.5,
        "leadingReactivePowerKvarh": 0.0,
        "laggingPowerFactor": 73.2,
        "leadingPowerFactor": 100.0,
        "nsm": 28800,
        "weekStatus": "Weekday",
        "dayOfWeek": "Monday",
        "loadType": "Maximum_Load"
      }
    ]
  }
}
```

后端说明：

- 后端从导入后的钢铁能源数据表或历史聚合表查询。

## 6. 核心评价指标口径

### 6.1 MAPE 仿真精度

字段名：`mape`

含义：平均绝对百分误差，数值越低越好。

前端展示时可转换为仿真精度：

```text
仿真精度 = 100% - MAPE
```

### 6.2 EC 轧钢能耗

字段名：`ec`

含义：单位合格产品能耗。

```text
EC = 生产总能耗 kgce / 合格产品产量 t
```

当前数据报告只包含能源相关字段，没有 `steelOutput` 和 `qualifiedOutput`。因此 EC 不能仅由该数据集直接计算，需要算法组补充产量数据，或由后端/算法组约定模拟产量数据。

### 6.3 ER 方案可执行率

字段名：`er`

含义：考核周期内可执行方案数 / 方案总数。

```json
{
  "executablePlanCount": 96,
  "totalPlanCount": 100,
  "er": 96.0
}
```

ER 依赖方案执行记录和约束校验结果，当前能源数据集不直接提供该字段，需要后端根据方案状态、冲突记录或算法输出计算。

| 方案主表和明细表 | 前端查询、报表统计 |
| MAPE/EC/ER 指标 | 方案评估、报表统计 |

### 7.2 算法结果校验

后端接收算法结果后，至少校验：

- 必填字段是否存在。
- `taskId` 是否一致。
- 时间序列是否按 `timeInterval` 连续。
- 当前数据集时间粒度是否保持 15 分钟/点。
- `startTime < endTime`。
- 产量、能耗、成本不能为负数。
- 负荷率不能超过合理范围。
- MAPE、EC、ER 是否可解析为数值。
- `timestamp` 源字段是否已成功解析，报告中的异常年份需先清洗。
- 日级排产 `schedule` 是否包含 24 条记录。
- 日级排产 `hour` 是否完整覆盖 0-23。
- 日级排产 `elec_forecast` 可选；若存在，检查是否基本满足 `production * elec_coefficient/EC_optimized`。

校验失败时：

```json
{
  "code": 602,
  "message": "算法返回结果校验失败",
  "data": null
}
```

## 9. 推荐联调流程

1. 算法组固定 `daily_plan` 和 `realtime_control` 两类 JSON 字段。
2. 后端通过导入接口保存真实排产和实时调控数据。
3. 前端优先联调登录、排产导入/详情、实时能源、实时调控 latest/history。
4. 能源运行方案、协同优化页面接后端派生计算接口完成展示闭环。
5. PPT 按“真实算法输出 + 后端基于已有数据派生计算”口径整理，不额外要求算法组补 `energy_plan.json` 和 `joint_optimization.json`。
6. 三方固化最终接口文档、数据库文档和指标口径。
